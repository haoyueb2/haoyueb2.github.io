写代码训练模型之前，有一个更重要的问题：**你的 GPU 装得下这个模型吗？训练要跑多久？** 如果答不上来就开跑，要么 OOM 崩掉，要么跑了三天发现参数设错了。

这就是 Chapter 3 的核心——**资源核算（Resource Accounting）**，也就是不跑代码就能估算训练成本的能力。

## 一、两个公式走天下

整章最值钱的就是两个公式。

### 公式 1：训练要多久？

$$总计算量 = 6 \times 参数量 \times Token 数$$

为什么是 6 倍？因为训练 = 前向传播 + 反向传播：

| 阶段 | 做了什么 | 计算量 |
|------|---------|-------|
| 前向传播 | 算预测值 | 2 × 参数量（乘法 + 加法） |
| 反向传播 | 算梯度 | 4 × 参数量（每层要算两个矩阵乘法） |
| **合计** | | **6 × 参数量** |

有了总计算量，除以硬件算力就得到时间：

$$训练时间 = \frac{6 \times 参数量 \times Token数}{GPU数量 \times 单卡算力 \times MFU}$$

其中 MFU（Model FLOPs Utilization）是实际利用率，一般按 50% 估算。

**实际算一把**：70B 模型，15T tokens，1024 张 H100（单卡 990 TFLOP/s）：

```
总 FLOPs = 6 × 70×10⁹ × 15×10¹² = 6.3 × 10²⁴
总算力   = 1024 × 990×10¹² × 50% = 5 × 10¹⁷ FLOP/s
时间     = 6.3×10²⁴ ÷ 5×10¹⁷ ≈ 1.26×10⁷ 秒 ≈ 146 天
```

五个月，还是不中断的理想情况。大模型训练真的是个巨大工程。

### 公式 2：需要多少显存？

训练时，显存里不只有模型参数，还有梯度和优化器状态：

| 内容 | 每参数占用 (FP32) | 说明 |
|------|------------------|------|
| 参数 | 4 bytes | 模型权重本身 |
| 梯度 | 4 bytes | 和参数一样大 |
| 优化器状态 | 8 bytes | AdamW 存一阶矩 + 二阶矩，各 4 bytes |
| **合计** | **16 bytes** | |

所以：

$$显存需求 = 参数量 \times 16\ bytes$$

**实际算一把**：8 张 H100（每张 80GB），FP32 AdamW，最大能训多大的模型？

```
总显存 = 8 × 80GB = 640GB
最大参数量 = 640GB ÷ 16 bytes = 40B
```

不是 160B（640 ÷ 4），是 **40B**。很多人第一次算都会忘记梯度和优化器的开销。

> 注意：这个估算没算激活值（activations）的内存占用，实际能训的模型还会更小一些。

## 二、浮点精度：省内存的关键

为什么上面算显存时说"FP32"？因为换一种精度，内存占用完全不同。训练大模型时选对数据类型，能省一半甚至更多的显存。

### 四种浮点类型

| 类型 | 大小 | 指数位 | 动态范围 | 精度 | 地位 |
|------|------|--------|---------|------|------|
| FP32 | 4 bytes | 8 位 | 大 | 高 | 默认，用于参数主副本和优化器 |
| FP16 | 2 bytes | 5 位 | **窄** | 中 | 已过时，容易溢出 |
| **BF16** | **2 bytes** | **8 位** | **大** | 低 | **当前主流** |
| FP8 | 1 byte | 4-5 位 | 小 | 极低 | 前沿研究，主要用于推理 |

### 为什么 BF16 赢了

FP16 和 BF16 都是 2 字节，但设计思路完全不同：

- **FP16**：保留了 10 位尾数（精度高），但指数位只有 5 位 → 能表示的数值范围很窄，训练时小梯度直接下溢变成 0，大数值上溢变成 Infinity
- **BF16**：只保留 7 位尾数（精度低），但指数位有 8 位（和 FP32 一样） → 动态范围和 FP32 一样宽

深度学习不在乎 3.14159 还是 3.14（精度），但非常在乎能不能表示 10⁻³⁸ 到 10³⁸ 这个范围。BF16 做了正确的取舍：**要范围，不要精度。**

### 混合精度训练

实际训练中不会只用一种精度，而是"混着来"：

- **前向传播的激活值**：用 BF16 → 省内存、算得快
- **参数主副本 + 梯度 + 优化器状态**：用 FP32 → 保证数值稳定

这就是混合精度训练（Mixed Precision Training），PyTorch 的 AMP（Automatic Mixed Precision）库可以自动处理精度转换。

## 三、反向传播的计算量为什么是前向的 2 倍

这个结论不难理解。以一个矩阵乘法 `h = x @ w` 为例：

**前向传播**只做了一件事：
```
h = x @ w    # FLOPs = 2BDK（B个样本，D维输入，K维输出）
```

**反向传播**要算两个梯度，每个都是一次矩阵乘法：
```
w.grad = x.T @ h.grad    # FLOPs = 2BDK（为了更新权重）
x.grad = h.grad @ w.T    # FLOPs = 2BDK（为了传给上一层）
```

反向做了两次矩阵乘法，所以是前向的 2 倍。前向 2 + 反向 4 = 6，这就是"6 倍法则"的来源。

## 四、训练循环：所有零件拼在一起

把前面的知识串起来，一个完整的训练循环其实只有 6 行核心逻辑：

```python
for step in range(num_steps):
    x, y = get_batch()           # 取一批数据
    pred = model(x)              # 前向传播
    loss = F.mse_loss(pred, y)   # 算损失
    loss.backward()              # 反向传播（算梯度）
    optimizer.step()             # 用梯度更新参数
    optimizer.zero_grad()        # 清空梯度，准备下一轮
```

几个工程细节：

**参数初始化很重要。** 直接用标准正态分布初始化权重，数值会随层数爆炸。Xavier 初始化通过除以 √(输入维度) 来保持数值稳定：

```python
w = nn.Parameter(torch.randn(input_dim, output_dim) / np.sqrt(input_dim))
```

**Checkpoint 是必须的。** 大模型训练动辄几个月，中途崩了不能从头来。检查点要保存两样东西：模型参数（`model.state_dict()`）和优化器状态（`optimizer.state_dict()`）。只存参数不存优化器状态的话，恢复训练时 Adam 的动量信息全丢了，训练效果会变差。

**数据加载可以异步。** 用 `pin_memory()` 把 CPU 张量锁在物理内存，再用 `non_blocking=True` 异步传输到 GPU，这样 GPU 算当前 batch 的同时 CPU 在准备下一个 batch，流水线起来吞吐量高很多。

## 五、GPU 的两个核心参数

对没接触过 GPU 编程的人来说，显卡参数可能看着头大。其实只需要关心两个数字：

| 参数 | 含义 | 类比 |
|------|------|------|
| **显存**（如 80GB） | 能装多少数据 | 服务器的 RAM |
| **算力**（如 990 TFLOP/s） | 每秒能做多少次运算 | CPU 核数 × 频率 |

显存决定"装不装得下"，算力决定"跑多快"。资源核算就是用这两个数字做除法。

主流 GPU 对比：

| GPU | 显存 | BF16 算力 | 常见场景 |
|-----|------|----------|---------|
| A100 | 80GB | 312 TFLOP/s | 上一代主力 |
| H100 | 80GB | 990 TFLOP/s | 当前主力 |

H100 比 A100 快了约 3 倍（在 BF16 下），但显存一样大。所以 H100 上跑得更快，但能装的模型大小没变。

## 六、用公式算算真实的模型

学了公式不用就会忘。拿几个真实模型练练手：

| 模型 | 参数量 | 训练 Token 数 | 估算总 FLOPs | 估算显存（FP32） |
|------|--------|-------------|-------------|---------------|
| GPT-2 | 124M | 约 10B | 7.4 × 10¹⁸ | ~2 GB |
| LLaMA 2 7B | 7B | 2T | 8.4 × 10²² | ~112 GB |
| LLaMA 2 70B | 70B | 2T | 8.4 × 10²³ | ~1120 GB |

几个有意思的观察：

- **GPT-2** 只要 2GB 显存，一张消费级显卡就能训。这也是为什么它经常被用作教学模型。
- **LLaMA 2 7B** 要 112GB，单张 H100（80GB）装不下。必须用模型并行或者混合精度（BF16 参数 + FP32 优化器）才行。
- **LLaMA 2 70B** 要 1.1TB，至少需要十几张 H100 才能放下参数和优化器状态，还没算激活值。

这就是为什么分布式训练（Chapter 8）和混合精度是大模型的标配——不是为了炫技，是物理上放不下。

## 七、一些容易忽略的点

**MFU 永远到不了 100%。** 理论峰值算力是硬件宣传数字，实际有内存读写、数据传输、同步等开销。MFU 能到 50% 已经算不错了，做资源估算时一定要打折。

**矩阵乘法占了 90%+ 的计算量。** LayerNorm、Softmax、激活函数等操作的 FLOPs 和矩阵乘法相比可以忽略不计。所以粗算时只数矩阵乘法就够了。

**内存里最大的消耗不一定是参数。** 对于大 batch size 或长序列，激活值的内存占用可能超过参数本身。这也是为什么后面会学到激活值检查点（Activation Checkpointing）等省内存技巧。

**精度选择直接影响内存和速度。** 同样的模型，BF16 比 FP32 省一半显存，而且在新 GPU 上算得更快。混合精度训练几乎是标配。
