最近我对一个自己的项目做了一次很大的减法。

它原来想做的是一条完整链路：抓内容，写入笔记库，再提供一套 CLI 去调用模型做问答。看起来很完整，也很“像产品”。但真正往前走时，我慢慢意识到，后半段很多复杂度其实不是来自真实需求，而是来自我自己额外造出来的一层系统。

这次调整，最后有两个很核心的结论。

第一，我并不需要专门做一个 CLI 去调用 Claude 或 Codex。

我最开始的思路是：既然已经有本地知识库，那就应该再包一层命令行，把 prompt、检索、上下文、输出格式都编排起来。这个想法听起来很合理，因为它有一种“闭环完成了”的满足感。

但问题也恰恰在这里。你一旦开始包这层 CLI，就会自然开始做更多事情：会话、日志、派生索引、上下文模式、特殊参数、失败恢复、输出协议。最后你会发现，自己不知不觉在做一个微型产品，而不是在解决“让 agent 更好地利用已有内容”这件事。

后来我反过来想，既然 agent 本来就已经很擅长遵循工作流，那为什么不把这层能力做成 skill？

skill 的好处不是“更轻”这么简单，而是它的抽象层级更对。它不要求你再造一个交互系统，也不要求你维护一套单独的运行时。你只需要把约束、流程、边界讲清楚，agent 就能直接用。它把复杂度留在通用能力里，而不是把复杂度固化成一个越来越重的专用壳。

这件事让我重新认识了 skill 的价值。之前我会把 skill 想成“说明书”或者“prompt 模板”。但它真正强大的地方，在于它可以承载稳定流程，而不一定需要承载专门代码。很多原本以为必须写成产品逻辑的东西，实际上用 skill 反而更优雅，因为它保留了灵活性，也减少了维护面。

第二，我也不需要一个专门的 Obsidian CLI 作为检索前提。

我之前一度很执着于“既然笔记最后在 Obsidian 里，那检索也应该走 Obsidian CLI”。这在概念上很统一，但实践里很快出现了一个更朴素的问题：为什么一定要这样？

如果 agent 最终面对的就是一堆 Markdown 文件，那么 `rg` 已经足够好。它快、稳定、透明、到处都能跑，而且没有额外协议。更重要的是，它没有把系统重新绑定回某个编辑器或某个工具品牌。

很多工程设计在一开始都会有一种“专用接口更优雅”的诱惑。你会觉得，既然有一个看起来语义更完整的入口，就应该围绕它设计系统。但我越来越觉得，真正的优雅往往不是多一层“语义正确”的包装，而是少一层不必要的依赖。

如果 `rg` 能解决问题，那它带来的好处非常直接：

1. 心智模型更简单。文件就是文件，搜索就是搜索，读取就是读取。
2. 系统边界更清晰。抓取层负责产出 Markdown，检索层直接面对 Markdown。
3. 可迁移性更强。今天你放在 Obsidian，明天也可以放在别的地方。
4. agent 的工作流更自然。先搜，再看，再总结，没有额外黑箱。

我后来总结，这次重构本质上不是“从一种技术换成另一种技术”，而是从“做一套第二系统”回到“直接利用已有基础设施”。

这是我最近越来越在意的一种工程哲学：做减法，不是偷懒，而是识别哪些复杂度是真问题，哪些复杂度只是工程师自己添加的仪式感。

很多时候，我们不是没有能力把事情做得更复杂，而是太容易把“我能做”误判成“我该做”。尤其是自己做个人项目时，这种误判更常见。因为你没有真实用户持续给你反馈，系统会自然朝着“更完整”“更可控”“更像产品”的方向膨胀。

但真正值得保留的系统，往往不是功能最多的那个，而是边界最清楚的那个。

抓取就是抓取。

写入就是写入。

检索就是检索。

问答就是在已有检索能力之上做组合，而不是再复制出一整套专用基础设施。

这次改完之后，我反而对整个项目更有信心了。不是因为它更“大”，而是因为它终于更像它自己了。它不再试图包办所有东西，而是把每一层都压回了更合适的位置。

工程里真正高级的能力，很多时候不是继续往上堆，而是知道哪一层应该删掉。
